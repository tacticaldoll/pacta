## Context

`openspec/changes/archive/` accumulates one dated directory per shipped change (22
today). Each holds a proposal/design/tasks plus a frozen copy of the delta spec
that was already synced into `openspec/specs/`. The living truth is
`openspec/specs/`; the release-facing narrative is `CHANGELOG.md`; identity is
`PROJECT.md`; durable "why not / reconsidered" decisions are BACKLOG's Recorded
Reconsiderations; and every diff and rich commit body is in git. The archive is a
belt-and-suspenders provenance store, redundant against git + specs + BACKLOG. This
change retires it (an explore-mode decision, greenlit).

## Goals / Non-Goals

**Goals:**
- Remove the archive and switch the lifecycle to sync-not-archive.
- Record the reversal honestly as an ADR superseding ADR-0002's archival clause.
- Keep every governance guarantee true after the archive no longer exists.

**Non-Goals:**
- No code, dependency, runtime, or published-surface change.
- No loss of decision history — git retains everything; BACKLOG keeps the durable
  reconsiderations.

## Decisions

### Sync-not-archive, agent-driven
This CLI has no `sync` command; its `archive` command is the only built-in that
syncs deltas into `openspec/specs/`, and it also moves the change to `archive/`. The
`openspec-sync-specs` skill is explicitly an **agent-driven** operation — read the
delta specs and edit `openspec/specs/` directly — gated by `openspec validate --all
--strict`. So the new terminal step is: sync the delta specs by hand into
`openspec/specs/`, validate, then `git rm -r` the change directory. No archive is
created. This change dogfoods that ritual on itself.

- Alternative considered: keep using `openspec archive` for its sync logic, then
  delete the archive entry it creates. Rejected — a transient archive entry plus a
  deletion is more awkward than an agent-driven sync, and manual sync for
  MODIFY/ADD deltas is low-risk under the strict validate gate. (Full-capability
  removal is the one fiddly case, and it already requires manual handling.)

### ADR discipline: supersede, don't rewrite
ADR-0002 (Accepted) decided OpenSpec-with-archive. ADRs are immutable records, so
ADR-0002's decision text is left as the historical record; a Status note points to
ADR-0004, and ADR-0004 records the retirement and its rationale. This keeps the
decision trail honest — the reversal is itself a recorded decision, matching the
project's governance identity.

### Governance scenarios stay true
Two scenarios (`governance-prose` "Prose Governance Reaction" and
`quality-governance` "Active Prose Governance") illustrate that historical prose is
not scanned, citing "archived OpenSpec changes." With the archive gone, that
illustration is stale, so both are reworded to cite git history and superseded ADRs.
The guarantee and the scan's behavior are unchanged — the prose scan already reads
only a fixed active-prose file list (`ACTIVE_PROSE_FILES` in the governance
constitution), never the archive — so no governance code changes.

### Out of scope: CLI shims and stale Purpose placeholders
Two archive references are deliberately left untouched:

- The OpenSpec CLI shims under `.claude/` and `.agent/` (e.g. `/opsx:archive`, the
  `openspec-archive-change` skill) still describe and perform archival. They are
  **generated by `openspec update`** (their frontmatter carries `generatedBy`), so
  hand-edits would be overwritten. The project's **authoritative** lifecycle is
  `AGENTS.md`, which after this change no longer archives; `/opsx:archive` remains a
  live way to recreate an archive but is off-ritual. Bringing the shims into line is
  an upstream/`openspec update` concern, not a hand-edit here.
- Six `openspec/specs/*/spec.md` files carry a stale `TBD - ... Update Purpose after
  archive` Purpose placeholder (public-facade, release-packaging, surface-tiers,
  contract-manifestation, lifecycle-persistence, registry-conformance). These were
  stale since those specs were created (a real Purpose was never written) — a
  defect independent of this change's thesis, and the Purpose is spec top-matter
  with no delta mechanism. They are handed to the release-readiness sweep rather
  than fixed via six out-of-thesis direct spec edits here.

## Risks / Trade-offs

- [A live "why not" exists only in an archived design.md] → Checked: BACKLOG's
  Recorded Reconsiderations hold the durable ones, and git history retains every
  archived file verbatim. Nothing is lost; only browsability-without-git changes.
- [Losing browsable in-tree provenance] → Accepted. The trade is deliberate:
  thinness over a redundant second copy of what git already holds.
- [Future manual sync drift] → The strict `openspec validate --all --strict` gate
  runs in the DoD and CI and catches a mis-synced spec.

## Migration Plan

Apply the doc/ADR edits and the two spec deltas, `git rm -r` the archive, sync the
deltas into `openspec/specs/` by hand, `openspec validate --all --strict`, run the
governance check, and land on `release/0.1.0`. Going forward, changes end at sync +
change-dir removal; no archive.
